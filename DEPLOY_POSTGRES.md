# النشر على Railway مع PostgreSQL

## 1. الإعداد الأول (تنصيب جديد بالكامل)

1. ادفع المشروع إلى Railway (من GitHub أو رفع مباشر).
2. من لوحة Railway: **New → Database → PostgreSQL** لإضافة خدمة PostgreSQL
   لنفس المشروع. Railway هيضيف تلقائياً متغير البيئة `DATABASE_URL`
   لخدمة التطبيق (Flask) — مفيش أي إعداد يدوي مطلوب.
3. متغيرات بيئة اختيارية يفضّل ضبطها من تبويب Variables:
   - `SECRET_KEY` — أي نص عشوائي طويل لتشفير الجلسات.
   - `DEVELOPER_USERNAME` / `DEVELOPER_PASSWORD` — بيانات دخول لوحة مطور
     النظام المخفية (`/system/developer-login`). لو متسيبتش، هيستخدم
     القيم الافتراضية الموجودة في `app.py` وده غير آمن للإنتاج.
4. Railway هيشغّل تلقائياً: `gunicorn app:app --bind 0.0.0.0:$PORT ...`
   (موجود في `Procfile` و `nixpacks.toml`).
5. عند أول تشغيل، السيرفر بينفّذ `main_db.create_all()` تلقائياً فينشئ
   جدولي `tenants` و `subscription_codes` في الـ schema `public` — مفيش
   أي أمر Migration يدوي مطلوب لأول مرة.
6. كل صالون بيسجّل بعد كده هيبقى ليه Schema منفصلة تلقائياً
   (`tenant_<slug>`) جوه نفس قاعدة PostgreSQL — بيتم إنشاؤها فور
   التسجيل (`create_tenant_schema`)، وأيضاً بشكل آمن (idempotent) في أي
   وقت لاحق عبر `ensure_tenant_schema` (Migration تلقائية لأي جداول
   جديدة تُضاف مستقبلاً لنفس البنية).

## 2. الترحيل من نسخة SQLite قديمة (لو عندك بيانات سابقة)

لو كنت شغّال بنسخة قديمة بتستخدم ملفات SQLite (`data/main.db` و
`data/tenants/*.db`) وعايز تنقل بياناتها للنسخة الجديدة على PostgreSQL:

1. تأكد إن `DATABASE_URL` مضبوط على قاعدة PostgreSQL **الجديدة والفاضية**.
2. ضع مجلد `data/` القديم (بنفس البنية: `data/main.db` و
   `data/tenants/<slug>.db`) بجانب `app.py` في بيئة التشغيل (أو حدد
   مساره عبر متغير البيئة `OLD_SQLITE_DATA_DIR`).
3. شغّل مرة واحدة فقط:
   ```bash
   python migrate_from_sqlite.py
   ```
4. السكربت هيقرأ كل المؤسسات من `main.db` القديم، وهينشئ لكل مؤسسة
   Schema مستقلة في PostgreSQL، وينسخ كل جداولها (المستخدمين، الإعدادات،
   الخدمات، العملاء، الحجوزات) مع الحفاظ على نفس الـ IDs، ثم يضبط
   الـ sequence تلقائياً عشان أي بيانات جديدة تكمل ترقيم صحيح.
5. السكربت آمن للتشغيل أكثر من مرة — أي سجل موجود بالفعل في PostgreSQL
   بيتم تجاهله (`ON CONFLICT DO NOTHING`).
6. بعد التأكد إن الترحيل نجح، احذف مجلد `data/` القديم نهائياً (مش
   محتاجه تاني، والنظام الجديد أصلاً مبيستخدمش أي ملفات محلية).

## 3. ملاحظات مهمة

- النظام دلوقتي **مايستخدمش SQLite إطلاقاً** — كل القراءة والكتابة
  بتتم على PostgreSQL فقط، سواء بيانات المؤسسات الرئيسية (`public`) أو
  بيانات كل صالون على حدة (`tenant_<slug>`).
- تأكد إن `DATABASE_URL` بيبدأ بـ `postgresql://` — لو Railway/Heroku
  رجّع القيمة بصيغة `postgres://` القديمة، الكود بيحوّلها تلقائياً.
- حذف مؤسسة من لوحة المطور بيمسح الـ Schema بالكامل من PostgreSQL
  (`DROP SCHEMA ... CASCADE`) — إجراء نهائي لا يمكن التراجع عنه.
